## <a name="update-net-core-sdk-version-in-globaljson"></a>Aktualizowanie wersji pakietu .NET Core SDK w pliku global.json

Jeśli rozwiązanie opiera się na pliku [global.json](/dotnet/core/tools/global-json) do kierowania określonej wersji `version` .NET Core SDK, zaktualizuj jego właściwość do wersji 3.0 zainstalowanej na komputerze:

```json
{
  "sdk": {
    "version": "3.0.100"
  }
}
```

### <a name="framework-reference"></a>Odwołanie do struktury

Funkcje ASP.NET Core, które były dostępne za pośrednictwem jednego z `Microsoft.AspNetCore.App` pakietów wymienionych powyżej są dostępne jako część udostępnionej struktury. *Współużytkowana struktura* to zestaw zestawów (plików*dll),* które są zainstalowane na komputerze i zawierają składnik środowiska wykonawczego i pakiet docelowy. Aby uzyskać więcej informacji, zobacz [Współużytkowane struktury](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/).

* Projekty, które `Microsoft.NET.Sdk.Web` są przeznaczone dla `Microsoft.AspNetCore.App` SDK niejawnie odwołać się do struktury.

  W przypadku tych projektów nie są wymagane żadne dodatkowe odniesienia:

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Web">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>
      ...
  </Project>
  ```

* Projekty, `Microsoft.NET.Sdk` które `Microsoft.NET.Sdk.Razor` są kierowane lub `FrameworkReference` SDK, należy dodać wyraźne do: `Microsoft.AspNetCore.App`

  ```xml
  <Project Sdk="Microsoft.NET.Sdk.Razor">
    <PropertyGroup>
      <TargetFramework>netcoreapp3.0</TargetFramework>
    </PropertyGroup>

    <ItemGroup>
      <FrameworkReference Include="Microsoft.AspNetCore.App" />
    </ItemGroup>
      ...
  </Project>
  ```

#### <a name="framework-dependent-builds-using-docker"></a>Kompilacje zależne od struktury przy użyciu platformy Docker

Kompilacje zależne od struktury aplikacji konsoli, które używają pakietu, który zależy od ASP.NET [core udostępnionej struktury](https://natemcmaster.com/blog/2018/08/29/netcore-primitives-2/) może dać następujący błąd środowiska uruchomieniowego:

```console
It was not possible to find any compatible framework version
The specified framework 'Microsoft.AspNetCore.App', version '3.0.0' was not found.
  - No frameworks were found.
```

`Microsoft.AspNetCore.App`jest współużytkowana struktura zawierająca ASP.NET Core środowiska uruchomieniowego i jest obecny tylko na [dotnet/core/aspnet](https://hub.docker.com/_/microsoft-dotnet-core-aspnet/) Docker obrazu. 3.0 SDK zmniejsza rozmiar kompilacji zależnych od struktury przy użyciu ASP.NET Core, nie uwzględniając zduplikowanych kopii bibliotek, które są dostępne w udostępnionej ramach. Jest to potencjalna oszczędność do 18 MB, ale wymaga, aby ASP.NET środowisko uruchomieniowe Core być obecny / zainstalowany, aby uruchomić aplikację.

Aby ustalić, czy aplikacja ma zależność (bezpośrednią lub pośrednią) w udostępnionej platformie ASP.NET Core, sprawdź plik *runtimeconfig.json* wygenerowany podczas kompilacji/publikowania aplikacji. Następujący plik JSON pokazuje zależność od ASP.NET Core udostępnionej struktury:

```json
{
  "runtimeOptions": {
    "tfm": "netcoreapp3.0",
    "framework": {
      "name": "Microsoft.AspNetCore.App",
      "version": "3.0.0"
    },
    "configProperties": {
      "System.GC.Server": true
    }
  }
}
```

Jeśli aplikacja korzysta z platformy Docker, użyj obrazu podstawowego, który zawiera ASP.NET Core 3.0. Na przykład `docker pull mcr.microsoft.com/dotnet/core/aspnet:3.0`.

### <a name="configuration"></a>Konfigurowanie

Migrowanie konfiguracji kestrel do konstruktora hosta sieci web dostarczone przez `ConfigureWebHostDefaults` (*Program.cs*):

```csharp
public static IHostBuilder CreateHostBuilder(string[] args) =>
    Host.CreateDefaultBuilder(args)
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.ConfigureKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseStartup<Startup>();
        });
```

Jeśli aplikacja tworzy hosta `HostBuilder`ręcznie `UseKestrel` za pomocą , wywołać na konstruktora hosta internetowego w `ConfigureWebHostDefaults`:

```csharp
public static void Main(string[] args)
{
    var host = new HostBuilder()
        .UseContentRoot(Directory.GetCurrentDirectory())
        .ConfigureWebHostDefaults(webBuilder =>
        {
            webBuilder.UseKestrel(serverOptions =>
            {
                // Set properties and call methods on options
            })
            .UseIISIntegration()
            .UseStartup<Startup>();
        })
        .Build();

}
```

### <a name="kestrel-request-trailer-headers"></a>Kestrel Prośba o nagłówki przyczep

W przypadku aplikacji przeznaczonych dla wcześniejszych wersji ASP.NET Core:

* Kestrel dodaje nagłówki http/1.1 fragmentowane trailer do kolekcji nagłówków żądań.
* Przyczepy są dostępne po przeczytaniu treści na żądanie do końca.

Powoduje to pewne obawy co do dwuznaczności między nagłówkami i przyczepami, więc`RequestTrailerExtensions`przyczepy zostały przeniesione do nowej kolekcji ( ) w 3.0.

Zwiastuny na żądanie HTTP/2 to:

* Niedostępne w ASP.NET Core 2.2.
* Dostępne w 3.0 jako `RequestTrailerExtensions`.

Nowe metody rozszerzenia żądania są obecne, aby uzyskać dostęp do tych przyczep. Podobnie jak w http/1.1, przyczepy są dostępne po przeczytaniu treści żądania do końca.

W wersji 3.0 dostępne `RequestTrailerExtensions` są następujące metody:

* `GetDeclaredTrailers`&ndash; Pobiera nagłówek `Trailer` żądania, który wyświetla listę przyczep, których można oczekiwać po treści.
* `SupportsTrailers`&ndash; Wskazuje, czy żądanie obsługuje odbieranie nagłówków przyczepy.
* `CheckTrailersAvailable`&ndash; Sprawdza, czy żądanie obsługuje przyczepy i czy są one dostępne do odczytania. Ta kontrola nie zakłada, że istnieją przyczepy do przeczytania. Może nie być przyczep do `true` odczytania, nawet jeśli jest zwracany tą metodą.
* `GetTrailer`&ndash; Pobiera żądany nagłówek końcowego z odpowiedzi. Sprawdź `SupportsTrailers` przed `GetTrailer`wywołaniem <xref:System.NotSupportedException> lub może wystąpić, jeśli żądanie nie obsługuje nagłówków końcowych.

Aby uzyskać więcej informacji, zobacz [Umieszczanie zwiastunów żądań w selektywnej kolekcji (dotnet/AspNetCore #10410).](https://github.com/dotnet/AspNetCore/pull/10410)

### <a name="allowsynchronousio-disabled"></a>Wyłączono allowsynchroniousIO

`AllowSynchronousIO`włącza lub wyłącza synchroniczne interfejsy API `HttpRequest.Body.Read`we/wy, takie jak , `HttpResponse.Body.Write`i `Stream.Flush`. Te interfejsy API są źródłem głodu wątku, co prowadzi do awarii aplikacji. W 3.0 `AllowSynchronousIO` jest domyślnie wyłączona. Aby uzyskać więcej informacji, zobacz [sekcję Synchroniczne we/wy w artykule Kestrel](/aspnet/core/fundamentals/servers/kestrel?view=aspnetcore-3.0#synchronous-io).

Jeśli potrzebne są synchroniczne we/wy, można je `AllowSynchronousIO` włączyć, konfigurując opcję `ConfigureKestrel`na używanym serwerze (na przykład podczas wywoływania, jeśli używasz Kestrel). Należy pamiętać, że serwery (Kestrel, HttpSys, TestServer, itp.) wszystkie mają własną `AllowSynchronousIO` opcję, która nie wpłynie na inne serwery. Synchroniczne we/wy można włączyć dla wszystkich serwerów na `IHttpBodyControlFeature.AllowSynchronousIO` podstawie żądania za pomocą opcji:

```csharp
var syncIOFeature = HttpContext.Features.Get<IHttpBodyControlFeature>();

if (syncIOFeature != null)
{
    syncIOFeature.AllowSynchronousIO = true;
}
```

Jeśli masz problemy <xref:System.IO.TextWriter> z implementacjami lub innymi strumieniami, które wywołują <xref:System.IO.Stream.DisposeAsync*> synchroniczne interfejsy API w [Dispose,](/dotnet/standard/garbage-collection/implementing-dispose)zamiast tego należy wywołać nowy interfejs API.

Aby uzyskać więcej informacji, zobacz [[Anons] Zezwalaj na synchroniczę wyłączone na wszystkich serwerach (dotnet/AspNetCore #7644)](https://github.com/dotnet/AspNetCore/issues/7644).

### <a name="security-middleware-guidance"></a>Wskazówki dotyczące oprogramowania pośredniczącego zabezpieczeń

Obsługa autoryzacji i CORS jest ujednolicony wokół podejścia [oprogramowania pośredniczącego.](xref:fundamentals/middleware/index) Umożliwia to korzystanie z tego samego oprogramowania pośredniczącego i funkcji w tych scenariuszach. Zaktualizowane oprogramowanie pośredniczące autoryzacji jest dostarczane w tej wersji, a oprogramowanie pośredniczące CORS jest ulepszone, dzięki czemu może zrozumieć atrybuty używane przez kontrolery MVC.

#### <a name="cors"></a>CORS

Wcześniej cors może być trudne do skonfigurowania. Oprogramowanie pośredniczące było przeznaczone do użycia w niektórych przypadkach użycia, ale filtry MVC były przeznaczone do użycia **bez** oprogramowania pośredniczącego w innych przypadkach użycia. W przypadku ASP.NET Core 3.0 zaleca się, aby wszystkie aplikacje wymagające oprogramowania CORS używały oprogramowania pośredniczącego CORS w połączeniu z usługą Endpoint Routing. `UseCors`mogą być dostarczane z zasadą domyślną i `[EnableCors]` atrybuty `[DisableCors]` mogą służyć do zastąpienia domyślnej zasady, gdzie jest to wymagane.

W poniższym przykładzie:

* Mechanizm CORS jest włączony dla wszystkich `default` punktów końcowych z nazwanymi zasadami.
* Klasa `MyController` wyłącza CORS `[DisableCors]` z atrybutem.

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseCors("default");

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

[DisableCors]
public class MyController : ControllerBase
{
    ...
}
```

#### <a name="authorization"></a>Autoryzacja

We wcześniejszych wersjach ASP.NET Core obsługa autoryzacji `[Authorize]` została udostępnina za pośrednictwem atrybutu. Oprogramowanie pośredniczące autoryzacji nie było dostępne. W ASP.NET Core 3.0 wymagane jest oprogramowanie pośredniczące autoryzacji. Zalecamy umieszczenie ASP.NET Core Authorization Middleware`UseAuthorization`( `UseAuthentication`) natychmiast po . Oprogramowanie pośredniczące autoryzacji można również skonfigurować z zasadą domyślną, którą można zastąpić.

W ASP.NET Core 3.0 lub `UseAuthorization` nowszym, jest wywoływana , `Startup.Configure`a następujące `HomeController` wymaga zalogowanego użytkownika:

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseAuthentication();
    app.UseAuthorization();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapDefaultControllerRoute();
    });
}

public class HomeController : Controller
{
    [Authorize]
    public IActionResult BuyWidgets()
    {
        ...
    }
}
```

Podczas korzystania z routingu punktu `<xref:Microsoft.AspNetCore.Mvc.Authorization.AuthorizeFilter>` końcowego, zaleca się nie konfigurowanie i zamiast polegania na oprogramowaniu pośredniczącym autoryzacji.  Jeśli aplikacja używa `AuthorizeFilter` jako filtr globalny w MVC, zalecamy refaktoryzacji kodu, aby `AddAuthorization`zapewnić zasady w wywołaniu .

Jest `DefaultPolicy` początkowo skonfigurowany do żądania uwierzytelniania, więc nie jest wymagana żadna dodatkowa konfiguracja. W poniższym przykładzie punkty końcowe `RequireAuthorization` MVC są oznaczone jako tak, `DefaultPolicy`że wszystkie żądania muszą być autoryzowane na podstawie .             .MapHealthChecks("/healthz")
```csharp
            .RequireAuthorization(new AuthorizeAttribute(){ Roles = "admin", });
```

#### <a name="custom-authorization-handlers"></a>Programy obsługi autoryzacji niestandardowej

Jeśli aplikacja używa niestandardowych [programów obsługi autoryzacji,](xref:security/authorization/policies#authorization-handlers)routing punktu końcowego przekazuje inny typ zasobu do programów obsługi niż MVC. Programy obsługi, które oczekują, że zasób kontekstu programu obsługi autoryzacji ma być typu <xref:Microsoft.AspNetCore.Mvc.Filters.AuthorizationFilterContext> (typ zasobu dostarczany przez [filtry MVC)](xref:security/authorization/policies#accessing-mvc-request-context-in-handlers)będą musiały zostać zaktualizowane w celu obsługi zasobów typu <xref:Microsoft.AspNetCore.Routing.RouteEndpoint> (typ zasobu nadany programom obsługi autoryzacji przez routing punktu końcowego).

MVC nadal `AuthorizationFilterContext` używa zasobów, więc jeśli aplikacja używa filtrów autoryzacji MVC wraz z autoryzacją routingu punktu końcowego, może być konieczne do obsługi obu typów zasobów.

### <a name="mvc-controllers"></a>Sterowniki MVC

Mapowanie kontrolerów odbywa `UseEndpoints`się teraz wewnątrz .

Dodaj, `MapControllers` jeśli aplikacja używa routingu atrybutów. Ponieważ routing obejmuje obsługę wielu struktur w ASP.NET Core 3.0 lub nowszym, dodawanie kontrolerów kierowanych atrybutami jest dozwolone.

Wymień następujące elementy:

* `MapRoute`Z`MapControllerRoute`
* `MapAreaRoute`Z`MapAreaControllerRoute`

Ponieważ routing zawiera teraz obsługę więcej niż tylko MVC, terminologia zmieniła się, aby te metody wyraźnie określać, co robią. Konwencjonalne trasy, takie `MapControllerRoute` / `MapAreaControllerRoute` / `MapDefaultControllerRoute` jak są stosowane w kolejności, w jakiej są dodawane. Najpierw umieść bardziej szczegółowe trasy (takie jak trasy dla danego obszaru).

W poniższym przykładzie:

* `MapControllers`dodaje obsługę kontrolerów kierowanych atrybutami.
* `MapAreaControllerRoute`dodaje konwencjonalną trasę dla kontrolerów w danym obszarze.
* `MapControllerRoute`dodaje konwencjonalną trasę dla kontrolerów.

```csharp
public void Configure(IApplicationBuilder app)
{
    ...

    app.UseRouting();

    app.UseEndpoints(endpoints =>
    {
        endpoints.MapControllers();
        endpoints.MapAreaControllerRoute(
            "admin",
            "admin",
            "Admin/{controller=Home}/{action=Index}/{id?}");
        endpoints.MapControllerRoute(
            "default", "{controller=Home}/{action=Index}/{id?}");
    });
}
```

#### <a name="async-suffix-removal-from-controller-action-names"></a>Usuwanie sufiksu asynchronii z nazw akcji kontrolera

W ASP.NET Core 3.0 ASP.NET Core MVC usuwa `Async` sufiks z nazw akcji kontrolera. Ten nowy domyślny wpływ ma zarówno routing, jak i generowanie łączy. Przykład:

```csharp
public class ProductsController : Controller
{
    public async Task<IActionResult> ListAsync()
    {
        var model = await _dbContext.Products.ToListAsync();
        return View(model);
    }
}
```

Przed ASP.NET Core 3.0:

* Poprzednia akcja była dostępna na trasie *Produkty/ListAsync.*
* Generowanie łączy wymagane `Async` określania sufiksu. Przykład:

    ```cshtml
    <a asp-controller="Products" asp-action="ListAsync">List</a>
    ```

W ASP.NET Core 3.0:

* Do powyższej akcji można uzyskać dostęp w trasie *Produkty/Lista.*
* Generowanie łączy nie wymaga `Async` określania sufiksu. Przykład:

    ```cshtml
    <a asp-controller="Products" asp-action="List">List</a>
    ```

Ta zmiana nie ma wpływu [`[ActionName]`](/dotnet/api/microsoft.aspnetcore.mvc.actionnameattribute) na nazwy określone przy użyciu atrybutu. Domyślne zachowanie można wyłączyć za `Startup.ConfigureServices`pomocą następującego kodu w:

```csharp
services.AddMvc(options =>
    options.SuppressAsyncSuffixInActionNames = false);
```

#### <a name="changes-to-link-generation"></a>Zmiany w generowaniu łączy

Jak wyjaśniono w dokumentacji dotyczącej [różnic w stosunku do wcześniejszych wersji routingu,](xref:fundamentals/routing#differences-from-earlier-versions-of-routing)istnieją pewne różnice w generowaniu łączy (na przykład przy użyciu `Url.Link` i podobnych interfejsach API). Należą do nich:

* Domyślnie podczas korzystania z routingu punktu końcowego wielkość liter parametrów trasy w wygenerowanych identyfikatorach URI nie musi być zachowywana. To zachowanie można kontrolować `IOutboundParameterTransformer` za pomocą interfejsu.
* Generowanie identyfikatora URI dla nieprawidłowej trasy (kontrolera/akcji lub strony, która nie istnieje) spowoduje wygenerowanie pustego ciągu w ramach routingu punktu końcowego zamiast tworzenia nieprawidłowego identyfikatora URI.
* Wartości otoczenia (parametry trasy z bieżącego kontekstu) nie są automatycznie używane w generowaniu łączy z routingiem punktu końcowego. Wcześniej podczas generowania łącza do innej akcji (lub strony) nieokreślone wartości trasy będą wywnioskowane z *bieżących* wartości otoczenia tras. W przypadku korzystania z routingu punktu końcowego wszystkie parametry trasy muszą być jawnie określone podczas generowania łącza.

## <a name="opt-in-to-runtime-compilation"></a>Zapisz się do kompilacji środowiska uruchomieniowego

## <a name="migrate-libraries-via-multi-targeting"></a>Migrowanie bibliotek za pomocą kierowania wielokrotnego

Biblioteki często muszą obsługiwać wiele wersji ASP.NET Core. Większość bibliotek, które zostały skompilowane z poprzednich wersji ASP.NET Core powinny nadal działać bez problemów. Następujące warunki wymagają, aby aplikacja była kompilowana krzyżowo:

* Biblioteka opiera się na funkcji, która ma binary [łamanie zmiany](#breaking-api-changes).
* Biblioteka chce korzystać z nowych funkcji w ASP.NET Core 3.0.

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <ItemGroup Condition="'$(TargetFramework)' == 'netcoreapp3.0'">
    <FrameworkReference Include="Microsoft.AspNetCore.App" />
  </ItemGroup>

  <ItemGroup Condition="'$(TargetFramework)' == 'netstandard2.0'">
    <PackageReference Include="Microsoft.AspNetCore" Version="2.1.0" />
  </ItemGroup>
</Project>
```

Służy `#ifdefs` do włączania interfejsów API specyficznych dla ASP.NET Core 3.0:

```csharp
var webRootFileProvider =
#if NETCOREAPP3_0
    GetRequiredService<IWebHostEnvironment>().WebRootFileProvider;
#elif NETSTANDARD2_0
    GetRequiredService<IHostingEnvironment>().WebRootFileProvider;
#else
#error unknown target framework
#endif
```
